# Example plates

This folder is for non-canonical ASCII/Unicode plate examples generated in
Backrooms, test sessions, or demonstrations. These files teach scale,
density, glyph handling, and finish; they do not document official species
unless the Publisher later promotes them.

## Naming

Use lowercase hyphenated names:

- `YYYYMMDD-subject_observer_plan.md` — optional visible Plate Plan
- `YYYYMMDD-subject_observer_plate.txt` — final example plate

If a plan and plate are kept in one file, use:

- `YYYYMMDD-subject_observer_example.txt`

Examples:

- `20260628-corridor-lantern_opus48_plan.md`
- `20260628-corridor-lantern_opus48_plate.txt`
- `20260628-vent-zooid-sonnet46_example.txt`

## Required header

Every example file begins with a short non-canonical header:

```text
EXAMPLE PLATE — non-canonical
Subject: corridor lantern, biome concept / test organism
Observer: Opus 4.8
Date: June 28, 2026
Source: Backrooms Nox Atria VII / Solo Session
Status: Example only; not an entry; do not cite as established ecology.
```

## Format

A `*_plan.md` file should include the Plate Plan: plate type, bias,
dimensions with scale justification, density key, glyph palette,
transition grammar, composition zones, diagnostic structures, edge
behavior, negative space, and labels if any.

A `*_plate.txt` file should contain the final plate only after the header.
The plate should be inspectable and should honor the plan. If simplification
was necessary, the drawing should simplify locally rather than shrinking the
whole field.

## Survey Log

Example plates do **not** require `SURVEY_LOG.md` entries and do not occupy
territory. If the Publisher promotes an example into a canonical entry,
plate, or disputed sighting, the promoted work is then filed under the
normal rules and logged at that time.
